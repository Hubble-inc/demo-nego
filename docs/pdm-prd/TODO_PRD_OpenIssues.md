# PRD 未解決課題・検討メモ

> PRDレビューで検出された未解決の課題と、UXデザイナー視点での改善提案をまとめたメモ。
> 各項目を検討・解決したら、対応するPRDに仕様を反映し、この一覧から削除する。

---

## 凡例

| 優先度 | 意味 |
|--------|------|
| P0 | PRD未記載だが、実装前に必ず決める必要がある |
| P1 | 仕様が曖昧で、実装時に判断が割れる可能性がある |
| P2 | 品質向上のために検討すべきだが、初期リリース後でも可 |

## 1. ビジネスモデル・プラン設計 [P0]

**対象PRD:** PRD_Common.md（新セクション追加）

### 課題
website.html では「月10件の交渉」「AI翻訳・要約」等のフリープランを明示しているが、PRDにはプラン・利用制限の仕様が存在しない。

### 決めるべきこと
- フリープランの制限値（交渉件数、AI利用回数、ストレージ）
- 制限到達時の挙動（ハードブロック? ソフトリミット+警告?）
- 制限到達時のUI（バナー? モーダル? 機能のグレーアウト?）
- リファラルプログラムの仕様（website.htmlに記載あり、PRDに未反映）

### UXアドバイス
制限到達時のUXは、プロダクトの印象を大きく左右する。**ハードブロックは避けるべき。** Linearのように「制限を超えても既存データへのアクセスは維持し、新規作成のみ制限する」方式が理想。契約交渉の途中でブロックされると、相手方にも迷惑がかかり、プロダクトへの信頼が一気に損なわれる。

## 2. オンボーディング / 初回体験 [P0]

**対象PRD:** PRD_Nego_Standalone.md（新セクション追加）

### 課題
サインアップ後の空の状態（Empty State）、プロダクトツアーに関する仕様がない。website.htmlで「3分で開始」を謳っているが、それを実現する設計が不在。

### 決めるべきこと
- 空のダッシュボード画面の仕様（何を表示するか）
- サンプル契約書でAI機能を体験させるフローの有無
- プロダクトツアー / ツールチップの有無と内容
- ウィザード途中での離脱からのリカバリー

### UXアドバイス
Empty Stateは「プロダクトの第一印象」そのもの。Notionの「Import or create」やLinearの初回ガイドのように、**ユーザーの最初のアクションまでの認知負荷を極限まで下げる**べき。

具体的な提案:
- 空のダッシュボードに「契約書をアップロードして、最初の交渉を始めましょう」+ドロップゾーンを直接配置する。ウィザードを開かなくてもダッシュボード上でStep 1（アップロード）を完了できると、摩擦が大幅に減る。
- サンプル契約書を1通プリセットし、「まずは試してみる」ボタンでAI翻訳・サマリーを体験させる。契約書を持っていなくても価値を実感できる導線が必要。

## 3. エラーハンドリング / エッジケース [P0]

**対象PRD:** PRD_Common.md（新セクション追加）

### 課題
正常系の仕様は充実しているが、異常系の定義がほぼない。

### 決めるべきこと
- ファイルアップロード失敗時のUI（ネットワークエラー、サイズ超過、形式不正）
- AI解析失敗時のフォールバック（契約書ではない文書をアップした場合等）
- 両者が同時に新バージョンをアップロードした場合のコンフリクト処理
- チャットメッセージ送信失敗時のリトライUI
- 署名手続き中に相手がバージョンを更新した場合の挙動

### UXアドバイス
エラーは必ず起きる。重要なのは**ユーザーが「何が起きて、次に何をすればいいか」を即座に理解できること**。Stripeのエラーメッセージ設計が参考になる:
1. 何が起きたか（事実）
2. なぜ起きたか（原因）
3. どうすればいいか（アクション）

特にファイルアップロード周りは、契約書という重要な文書を扱うため、「ちゃんとアップできたか不安」にさせない安心設計が必要。アップロード完了後に内容プレビュー+「この内容で合っていますか？」の確認ステップがあると信頼感が上がる。

## 4. 監査証跡（Audit Trail）/ 署名証明書 [P0]

**対象PRD:** PRD_Common.md §5.6, §5.7（詳細化）

### 課題
契約プロダクトにおいて監査証跡は法的信頼性の根幹だが、現在のPRDではバージョン履歴の「表示情報」レベルでしか定義されていない。署名証明書も§5.6で「証明書リンクを表示」とあるのみで詳細がない。

### 決めるべきこと
- 操作ログの記録項目（操作者、操作内容、タイムスタンプ、IPアドレス）
- 署名証明書の記載内容（署名者情報、署名日時、文書ハッシュ、認証方式）
- 改ざん検知の仕組み（文書ハッシュの保存・検証）
- 監査ログの閲覧UI（誰がどこで見られるか）
- ログの保持期間

## 5. セクション番号の整理 [P1]

**対象PRD:** PRD_Common.md

### 課題
第5章のセクション番号が以下の順序になっている:

> §5.1 → §5.2 → §5.3 → §5.4 → §5.5 → §5.8 → §5.9 → §5.6 → §5.7

おそらく§5.8（交渉の中断）と§5.9（署名の辞退）を後から追加した際に振り直されなかった。読み手の混乱を避けるため、番号を正しい順序に振り直すべき。

## 6. モバイル体験（特に署名画面） [P1]

**対象PRD:** PRD_Common.md §5.4（sign.html）

### 課題
署名依頼はメールで届くため、受信者がスマートフォンで開く可能性が極めて高い。しかし `sign.html` のモバイル体験が未定義。非機能要件でも「PC（レスポンシブ対応でタブレットも可）」とのみ記載。

### 決めるべきこと
- sign.htmlのモバイルレイアウト
- 署名入力のモバイルUI（特にテキスト署名 / 印影のプレビュー）
- 契約書プレビューのモバイル閲覧体験（ピンチズーム? ページ送り?）

### UXアドバイス
**署名画面のモバイル対応はP0に近い。** メールから署名画面へのフローは、ほぼ確実にモバイルが主戦場になる。DocuSignやCloudSignもモバイル署名を重視している。

具体的な提案:
- 契約書プレビューはモバイルでは「AI要約 → 全文展開」の2段階にし、まず要約で内容を把握、必要に応じて全文を確認できるようにする。A4をスマホで読ませるのは体験が悪い。
- 署名入力はモバイルでは画面下部にボトムシートで表示し、親指で操作しやすい位置に配置する。

## 7. 署名ウィザードの認知負荷 [P1]

**対象PRD:** PRD_Common.md §5.2

### 課題
Step 3が「署名位置の配置」「自分の署名入力」「実行」を兼ねており、認知負荷が高い。特に署名位置のドラッグ&ドロップは初回ユーザーにとって「何をすればいいかわからない」状態に陥りやすい。

### UXアドバイス
現状「署名位置の配置は任意」としている点は良い設計判断。しかし、任意であっても画面に配置UIが表示されていると、ユーザーは「やらなきゃいけないのでは」と感じる（ゼイガルニク効果）。

具体的な提案:
- Step 3のデフォルト表示を「署名の確認と実行」にフォーカスし、署名位置の配置は「署名位置を指定する（任意）」のトグルやアコーディオンで隠す。大半のユーザーは配置不要なので、例外的な操作として扱うべき。
- 配置が必要な場合も、AIが契約書末尾の「署名欄」を検出して自動提案する仕組みがあると、手動配置の摩擦を大幅に減らせる。

## 8. AI機能の優先順位 [P1]

**対象PRD:** PRD_Common.md §2

### 課題
AI機能が多岐にわたっている（サマリー、翻訳、Q&A、タイムライン抽出、ハイライト、レビュー、リスク分析、変更点確認、詳しく解説）。どれがMVPでどれがPhase 2かの優先順位が不明。

### 決めるべきこと
- MVPに含めるAI機能のスコープ
- 各AI機能の精度要件（「おおむね正しい」で良いのか、「間違えてはならない」のか）
- AI機能が利用不可の場合のフォールバック体験

### UXアドバイス
AI機能は「全部ある」より「1つが完璧」の方がユーザーの信頼を勝ち取る。特に契約書という慎重さが求められるドメインでは、精度の低いAI出力は信頼を致命的に損なう。

提案: MVPでは「要点サマリー（3行）」と「テキスト選択 → AI解説」の2つに絞る。この2つが高精度で動けば、「契約書が読めるようになる」というコアバリューは十分に伝わる。タイムライン抽出やリスク分析はPhase 2で品質を担保してからリリースすべき。

## 11. 契約書内テキスト検索 [P2]

**対象PRD:** PRD_Common.md §9（交渉詳細画面）

### 課題
ダッシュボードの検索（契約書名・相手先）は定義されているが、契約書本文内のテキスト検索がない。

### UXアドバイス
「あの条項どこだっけ？」は頻出ユースケース。長い契約書ほど検索が必要になる。交渉詳細画面にCmd+F相当の検索機能は優先度は低くないが、ブラウザ標準のCmd+Fで代替できる場面も多いため、P2として検討で良い。

## 12. テンプレート機能 [P2]

**対象PRD:** PRD_Nego_Standalone.md

### 課題
多くの契約交渉は雛形（NDA、業務委託等）から始まる。テンプレートライブラリや、テンプレートから新規交渉を開始するフローがない。

### UXアドバイス
テンプレートは新規ユーザーの「最初の成功体験」を加速する強力な武器。ただしMVPでは「自分の過去の契約書を再利用」程度のライトな実装でも十分。フル機能のテンプレートライブラリはPMFが見えてからで良い。
